# 导航系统核心研发：项目分工与实践指南

## 1. 题目要求

车载导航系统是在城市中驾车的好帮手，不仅能够计算出到达目的地的最优路线，而且可以显示当前位置附近的一些附加信息，比如附近的加油站，餐馆等等。请设计一款导航软件，实现导航系统的核心功能。

### 目标用户
- 汽车司机。

### 数据配置
导航系统中的地图可以抽象为一个图，地点信息和路径信息可以抽象为图中的顶点和边。请设计算法，来产生模拟的地图数据：
- a) 随机产生N个二维平面上的顶点（N>=10000），每个顶点对应地图中的一个地点
- b) 对于每个地点x，随机建立若干个连边到x附近的地点。每条连边代表一条路径，路径的长度等于边的两个顶点之间的二维坐标距离。
- c) 模拟数据必须保证，产生的图是一个连通图，并且道路之间不应有不合理的交叉。

### 功能要求
- **F1. 地图显示功能。** 输入一个坐标，显示距离该坐标最近的100个顶点以及相关联的边。关于如何用vc++画图，或java画图，请在baidu或google上查找相关方法，或者借阅相关的参考书。
- **F2. 地图缩放功能。** 提示：地图缩的越小，屏幕上显示的点数就越多，但是太多的点，会看不清楚。所以可以考虑只选择一个单元区域内只显示一个代表的点。
- **F2. 任意指定两个地点A和B**，能计算出A到B的最短路径。并将该最短路径经过的顶点以及连边显示出来。
- **F3. 模拟车流。** 请为每条连边增加两个属性：车容量v（饱和状态下这条路所能容纳的汽车的数量）、当前在这条路上的车辆数目n（n>v时为超负荷运作）。假设该路的长度为L，则该路的通行时间可模拟为`cLf(n/v)`，其中c是常数；`f(x)`是一个分段函数, x小于等于某个常数时，`f(x) = 1`，当x大于该常数时，`f(x) = 1+e^x`。每条道路的车容量v和道路的长度L为预先指定的固定参数。请模拟产生汽车在地图中行驶，为简化模型假设在同一条路上每架汽车穿越该路的时间均等于`cLf(n/v)`。要求实现模拟车流的动态变化，任意时刻，给定一个坐标，能在界面上显示该坐标附近的所有路径，并动态显示各个路径上的车流量的大小（可用不同颜色或其他方法区分车流量的大小级别）。
- **F4. 综合考虑路况的最短路径。** 任意时刻，指定两个地点A和B，能根据当前的路况，计算出从A到B最短行车时间，以及相应的最佳路径，并在界面上，将该最短路径经过的顶点以及连边显示出来。

### 项目目标
实现支持 10000 节点的城市导航模拟系统，包含地图生成、空间查找、动态车流模拟及动态最短路径规划。

### 协作模式
接口驱动开发（API-Driven Development）。所有人基于约定好的函数接口独立开发，互不干扰。

## 一、成员 A（算法核心负责人）

### 定位
负责所有与图论搜索和最短路径相关的核心计算。

### 负责模块
- **静态最短路径规划（对应功能 F2）：** 实现经典的最短路径算法（如 Dijkstra 或 A* 算法），计算任意两点间的最短距离。
- **动态路况最短路径（对应功能 F4）：** 改造上述算法，使其能够读取道路的实时拥堵权重，计算“行车时间最短”的最优解。

### 核心数据结构与算法（报告高分点）
- 带权有向/无向图的底层实现（邻接表，节省内存）。
- 优先队列（最小堆）优化 Dijkstra 算法，将时间复杂度降至 `O(V log V + E)`。
- 启发式搜索策略（A* 算法）的探索与应用。

### 实践指南与避坑提示
- **独立测试：** 前期不需要依赖界面的完成度。只需在控制台构建包含少数节点的小型测试图，验证最短路径算法逻辑的正确性，确保能正确输出路径节点的 ID 数组即可。
- **权重隔离：** 编写算法时，将“计算两点间距离”和“计算两点间通行时间”封装为两个独立的回调函数或方法。在 F2 和 F4 模式切换时，仅需替换权重计算逻辑，无需修改寻路算法主体。

## 二、成员 B（空间与动态引擎负责人）

### 定位
负责处理坐标空间的快速搜索以及复杂的路况数学建模。

### 负责模块
- **周边坐标检索（对应功能 F1）：** 给定鼠标点击的坐标，从 10000 个点中快速找出距离最近的 100 个点。
- **车流动态数学模拟（对应功能 F3）：** 根据任务书的数学公式 `cLf(n/v)`，编写路况更新引擎，定时刷新每条道路的车流量和拥堵状态。
- **地图缩放过滤算法（对应功能 F2 缩放）：** 当地图缩小时，按一定策略决定哪些节点该隐藏或抽样显示。

### 核心数据结构与算法（报告高分点）
- **Top-K 空间搜索算法**（从基础的距离排序起步，进阶可实现 KD-Tree 或四叉树，以提升检索效率）。
- 复杂分段函数 `f(x) = 1+e^x` 的代码抽象与动态状态更新（状态机模式）。

### 实践指南与避坑提示
- **数据抽样策略：** 在实现缩放功能时，若全量绘制 10000 个点会导致界面卡顿。建议将地图划分为网格（Grid），缩小时每个网格仅返回 1 个代表性节点交由前端绘制。

## 三、成员 C（图形交互与底层数据负责人）

### 定位
负责生成底层图数据、文件读写、提供数据访问接口，以及最终的界面集成与展示。

### 负责模块
1.  **地图数据生成与属性定义：**
    - 随机生成 10000 个点的坐标（ID, x, y）。
    - 为每个节点连接到其最近的k个邻居，以模拟真实的城市路网结构，取代模糊的“避免不合理交叉”要求。
    - 通过算法（如DFS/BFS）确保最终生成的图是完全连通的。
    - 为每条边生成核心属性：道路长度 L (欧氏距离) 和车容量 v (在合理范围内随机生成，如 100-500)。

2.  **数据持久化与格式规范：**
    - 将生成的地图数据（节点和边）序列化为统一的 JSON 格式并存入本地文件，供团队共享。格式定义如下：
      ```json
      {
        "nodes": [
          {"id": 0, "x": 12.3, "y": 45.6},
          ...
        ],
        "edges": [
          {"source": 0, "target": 1, "length": 15.2, "capacity": 300},
          ...
        ]
      }
      ```
    - 实现程序启动时反序列化加载地图文件的功能。

3.  **GUI 界面搭建与交互：**
    - **基础渲染：** 使用可视化库（如 EasyX、Qt 或 Java GUI）绘制地图的节点和边。
    - **交互处理：** 捕捉并响应鼠标事件，如点击选择路径规划的起点和终点，支持界面拖拽和缩放。
    - **动态可视化：** 根据成员 B 提供的接口获取的拥堵信息，实时以不同颜色（如绿、黄、红）渲染道路，并高亮显示成员 A 计算出的最短路径。

4.  **数据访问接口实现：**
    - 在 `NavigationAPI` 中，实现 `loadMap` 函数。
    - 同时，实现供成员 A 和 B 调用的数据访问函数，例如 `getNodeById(int id)`、`getEdge(int source, int target)`、`getAllNodes()` 等，将底层数据结构对其他成员透明化。

### 核心数据结构与算法（报告高分点）
- 图的连通性保证算法（最小生成树 + 随机加边）。
- 高效的 JSON 序列化/反序列化方案。
- 图形界面的事件驱动架构与状态管理。

### 实践指南与避坑提示
- **图的连通性生成技巧：** 可先通过“最小生成树（Prim 或 Kruskal）”或“随机游走”生成包含所有节点的连通骨架，再随机添加额外的边以增加路径选择。
- **面向“假数据”编程（Mocking）：** 在成员 A 的寻路算法完成前，为了推进界面的画线功能测试，可直接在寻路接口中硬编码返回一组固定的节点 ID（如返回节点 1、5、8）。只要界面能成功将这三个点连线，UI 开发任务即可独立闭环。待算法完成后替换为真实接口即可。

## 四、核心沟通桥梁：接口定义规范 (NavigationAPI)

```cpp
// 1. 基础数据结构定义

// Point 结构体: 包含节点 ID (int)，横坐标 x (double)，纵坐标 y (double)。
struct Point {
    int id;
    double x;
    double y;
};

// PathResult 结构体: 包含途径节点 ID 序列 (vector<int>)，以及总花费/时间 (double)。
struct PathResult {
    std::vector<int> node_ids;
    double total_cost;
};

// 2. 成员 C 提供的接口 (底层数据加载)

// 生成地图并存盘
bool generateAndSaveMap(int nodeCount, std::string filePath); 

// 读取地图文件到内存
bool loadMap(std::string filePath); 

// 3. 成员 B 提供的接口 (空间搜索与动态车流)

// 获取距离指定坐标最近的 count 个点
std::vector<Point> getNearestPoints(double x, double y, int count); 

// 更新全图的交通状况，每次调用刷新一次车流状态
void updateTrafficStatus(); 

// 获取某条边的当前拥堵权重（供成员 A 及成员 C 调用）
double getEdgeTrafficWeight(int edgeId); 

// 4. 成员 A 提供的接口 (核心寻路引擎)

// 计算最短路径（considerTraffic 为 false 计算最短距离，为 true 计算最短时间）
PathResult calculateShortestPath(int startId, int endId, bool considerTraffic); 
```

## 六、推荐项目文件结构 (C++/Qt)

对于一个使用C++和Qt的项目，一个良好、清晰的文件结构至关重要，尤其是在团队协作中。它能帮助每个人明确自己的工作范围，并使代码更易于管理和维护。

推荐一个基于“关注点分离”思想的经典项目结构：

```
City-Navigation-System/
├── CMakeLists.txt         // [构建] 项目构建配置文件 (或 .pro 文件)
│
├── 3rdparty/              // [外部库] 用于存放第三方库
│   └── nlohmann_json/
│       └── json.hpp       // (例如，一个用于解析JSON的库)
│
├── data/                  // [数据] 存放生成的地图文件
│   └── map.json
│
├── src/                   // [源码] 所有C++源码的根目录
│   │
│   ├── main.cpp             // [入口] 应用程序的入口点
│   │
│   ├── api/                 // [接口] 团队协作的核心，定义公共接口和数据结构
│   │   └── NavigationAPI.h  // 所有成员都需要遵守的“契约”
│   │
│   ├── core/                // [核心] 存放所有非GUI的核心业务逻辑
│   │   ├── DataModel/       // (可选) 将数据结构和模型再分组
│   │   │   ├── Graph.h      // (C) 定义图的数据结构、加载/保存、提供数据访问
│   │   │   └── Graph.cpp
│   │   │
│   │   ├── modules/
│   │   │   ├── DataGenerator.h    // (C) 负责生成地图数据
│   │   │   ├── DataGenerator.cpp
│   │   │   ├── PathFinder.h       // (A) 负责路径规划算法
│   │   │   ├── PathFinder.cpp
│   │   │   ├── SpatialSearch.h    // (B) 负责空间检索
│   │   │   ├── SpatialSearch.cpp
│   │   │   └── TrafficSimulator.h // (B) 负责模拟交通流
│   │   │   └── TrafficSimulator.cpp
│   │
│   └── gui/                 // [界面] 存放所有Qt图形界面相关的代码
│       │
```

### 各文件/目录的作用详解：

#### 顶层目录
*   `CMakeLists.txt` (或 `.pro` 文件): **项目“蓝图”**。它告诉编译器如何编译和链接您的整个项目，包括需要哪些源文件、依赖哪些Qt模块（如 `Core`, `GUI`, `Widgets`）以及如何包含第三方库。
*   `3rdparty/`: **“工具箱”**。用于存放您项目中用到的、但非您自己编写的第三方代码。例如，成员C需要处理JSON文件，可以将一个轻量级的JSON库（如 `nlohmann/json`）放在这里。
*   `data/`: **“仓库”**。用于存放您的程序运行时需要读写的数据文件，比如您生成的 `map.json`。这能让代码和数据分离，保持项目整洁。
*   `src/`: **“源代码核心区”**。您和团队成员编写的所有 `.h` 和 `.cpp` 文件都存放在这里。

#### `src` 源码目录
*   `main.cpp`: **“启动器”**。这是整个程序的入口，它的职责很简单：创建Qt应用实例 (`QApplication`) 和主窗口 (`MainWindow`)，然后把它显示出来。
*   `api/NavigationAPI.h`: **“团队协作的契约”**。这是项目中最关键的文件之一。它定义了所有团队成员共享的数据结构（如 `Point`, `PathResult`）和模块间的调用接口（如 `calculateShortestPath`, `getNearestPoints`）。所有人都面向这个文件编程，而不是具体实现。
*   `core/`: **“项目的大脑”**。存放所有与计算、处理和数据管理相关的非界面代码。
    *   `Graph.h/.cpp` (成员C): **数据核心**。这个类用于在内存中表示地图。您将在这里实现加载/保存JSON文件的功能，并提供公共函数（如 `getNodeById`）让成员A和B能够安全地访问地图数据。
    *   `DataGenerator.h/.cpp` (成员C): **数据生成器**。负责实现地图数据的随机生成、保证连通性等逻辑。
    *   `PathFinder.h/.cpp` (成员A): **寻路引擎**。实现Dijkstra或A*算法，它会通过 `Graph` 类提供的接口来获取地图信息。
    *   `SpatialSearch.h/.cpp` & `TrafficSimulator.h/.cpp` (成员B): **空间和动态引擎**。实现周边检索和交通流模拟。
*   `gui/`: **“项目的脸面”**。存放所有与用户看得到、摸得着的界面相关的代码。
    *   `MainWindow.h/.cpp/.ui`: **主窗口框架**。负责创建和管理窗口、菜单栏、工具栏、状态栏以及各个UI控件。它是所有GUI元素的“总指挥”。
    *   `MapView.h/.cpp` (成员C的核心GUI工作): **地图画布**。这是一个自定义的Qt控件（通常继承自 `QWidget`），您将在这里完成所有核心的绘制工作。
        *   **绘制**: 重写 `paintEvent()` 函数，使用 `QPainter` 将 `Graph` 中的节点和边画在屏幕上。
        *   **交互**: 重写鼠标事件（如 `mousePressEvent`）来获取用户点击坐标，用于选择起点/终点，并实现地图的拖拽和缩放。
        *   **更新**: 调用其他模块的接口，根据返回的路径和交通数据，动态地改变线条颜色或高亮路径。
